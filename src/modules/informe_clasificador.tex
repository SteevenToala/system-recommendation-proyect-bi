\section{Introducci\'on}
% =============================================================================

El presente informe explica la implementaci\'on del m\'odulo de preparaci\'on de
datos para Machine Learning dentro del proyecto de Sistema de Recomendaci\'on de
Alquileres de Pel\'iculas, desarrollado en la materia de \textbf{Inteligencia
de Negocios}.

El m\'odulo est\'a compuesto por tres archivos Python que trabajan en cadena,
cada uno con una responsabilidad bien definida:

\begin{enumerate}
  \item \textbf{\texttt{consultas\_clasificador.py}} --- Se conecta a MongoDB
        y extrae los datos mediante pipelines de agregaci\'on.
  \item \textbf{\texttt{dataframes\_bi.py}} --- Centraliza y limpia los
        DataFrames para que est\'en listos en cualquier parte del proyecto.
  \item \textbf{\texttt{clasificador\_binario.py}} --- Crea las variables
        derivadas y las etiquetas que usar\'an los modelos de ML.
\end{enumerate}

\begin{cajainfo}{Arquitectura de dependencias}
\centering
\texttt{clasificador\_binario.py}
$\longrightarrow$ \texttt{dataframes\_bi.py}
$\longrightarrow$ \texttt{consultas\_clasificador.py}
$\longrightarrow$ MongoDB (\texttt{BI\_Final.alquileres})
\end{cajainfo}

% =============================================================================
\section{Archivo 1: \texttt{consultas\_clasificador.py}}
% =============================================================================

\subsection{Prop\'osito}

Este archivo es el \'unico punto de contacto con la base de datos.
Su responsabilidad es \textbf{conectarse a MongoDB y retornar DataFrames
limpios}, sin que el resto del proyecto necesite conocer los detalles
de la conexi\'on ni de la estructura de la colecci\'on.

\subsection{Conexi\'on a MongoDB}

\begin{lstlisting}[caption={Par\'ametros de conexi\'on}]
MONGO_URI = "mongodb://localhost:27017/"
DB_NAME   = "BI_Final"
COLECCION = "alquileres"

def _conectar():
    client = MongoClient(MONGO_URI)
    return client[DB_NAME][COLECCION]
\end{lstlisting}

\subsection{Consultas implementadas}

\subsubsection{Consulta 1 --- Alquileres (tabla de hechos)}

Trae un registro por alquiler con los cinco campos necesarios para ML.
El pipeline usa \texttt{\$project} para seleccionar \'unicamente las
columnas relevantes.

\begin{table}[H]
  \centering
  \caption{Campos de \texttt{consulta\_alquileres()}}
  \begin{tabular}{lll}
    \toprule
    \textbf{Campo}            & \textbf{Tipo}  & \textbf{Descripci\'on} \\
    \midrule
    \texttt{cliente\_ref}      & string  & Identificador del cliente \\
    \texttt{pelicula\_ref}     & string  & Identificador de la pel\'icula \\
    \texttt{pelicula\_titulo}  & string  & T\'itulo legible \\
    \texttt{categoria\_nombre} & string  & G\'enero o categor\'ia \\
    \texttt{ingreso}           & float   & Monto cobrado \\
    \bottomrule
  \end{tabular}
\end{table}

El operador \texttt{\$ifNull} permite que la consulta funcione aunque los
documentos tengan estructuras distintas (el campo cliente puede llamarse
\texttt{id\_cliente}, \texttt{cliente.id\_cliente} o
\texttt{dimensiones.cliente.id\_cliente}).

\begin{lstlisting}[caption={Fragmento del pipeline de alquileres}]
pipeline = [
    {
        "$project": {
            "_id": 0,
            "cliente_ref": {
                "$toString": {
                    "$ifNull": [
                        "$id_cliente",
                        "$cliente.id_cliente",
                        "$dimensiones.cliente.id_cliente"
                    ]
                }
            },
            "ingreso": {"$ifNull": ["$ingreso", 0]}
        }
    }
]
\end{lstlisting}

\subsubsection{Consulta 2 --- Cat\'alogo de pel\'iculas}

Retorna una fila por pel\'icula (sin duplicados) usando \texttt{\$group}.
\'Util para mostrar metadatos del cat\'alogo disponible.

\subsubsection{Consulta 3 --- Clientes \'unicos}

Retorna el listado de clientes que alguna vez realizaron un alquiler,
permitiendo conocer el universo total de clientes del negocio.

\subsection{Funci\'on principal}

\begin{lstlisting}[caption={Funci\'on que ejecuta las tres consultas}]
def cargar_dataframes() -> dict:
    df_alquileres = consulta_alquileres()
    df_peliculas  = consulta_peliculas()
    df_clientes   = consulta_clientes()
    if df_alquileres.empty:
        raise ValueError("No hay datos en MongoDB.")
    return {
        "df_alquileres": df_alquileres,
        "df_peliculas":  df_peliculas,
        "df_clientes":   df_clientes,
    }
\end{lstlisting}

\subsection{Limpieza del campo \texttt{ingreso}}

El campo \texttt{ingreso} llegaba desde MongoDB en formatos inconsistentes,
produciendo valores err\'oneos como \texttt{38.559.032.846.715.300} al
interpretar puntos como separadores de miles en lugar de decimales.

La funci\'on \texttt{\_limpiar\_ingreso()} normaliza el valor seg\'un el
formato detectado:

\begin{table}[H]
  \centering
  \caption{Formatos que maneja la funci\'on de limpieza}
  \begin{tabular}{lll}
    \toprule
    \textbf{Valor original} & \textbf{Formato detectado}      & \textbf{Resultado} \\
    \midrule
    \texttt{"38.559"}       & Punto decimal europeo           & \texttt{38.559} \\
    \texttt{"38,559.00"}    & Coma de miles, punto decimal    & \texttt{38559.00} \\
    \texttt{"38.559.032"}   & M\'ultiples puntos = miles      & \texttt{38559.032} \\
    \texttt{38.559}         & Ya es float                     & \texttt{38.559} \\
    \texttt{null / NaN}     & Valor nulo                      & \texttt{0.0} \\
    \bottomrule
  \end{tabular}
\end{table}

\begin{lstlisting}[caption={L\'ogica de limpieza de ingreso}]
def _limpiar_ingreso(valor) -> float:
    if pd.isna(valor):
        return 0.0
    if isinstance(valor, (int, float)):
        return float(valor)
    s = str(valor).strip().replace("$","").replace(" ","")
    if "," in s and "." in s:
        s = s.replace(",", "")
    elif s.count(".") > 1:
        partes = s.rsplit(".", 1)
        s = partes[0].replace(".", "") + "." + partes[1]
    try:
        return float(s)
    except ValueError:
        return 0.0
\end{lstlisting}

\section{Archivo 3: \texttt{clasificador\_binario.py}}

\subsection{Prop\'osito}

    \toprule
    \textbf{Variable}                   & \textbf{Descripci\'on} \\

    \texttt{veces\_alquilada\_pelicula}  & Cantidad de veces que fue alquilada cada pel\'icula. \\
    \texttt{precio\_promedio\_genero}    & Ingreso promedio de todos los alquileres de ese g\'enero. \\

\subsection{Etiquetas binarias}

Las etiquetas toman el valor \textbf{0} (No) o \textbf{1} (S\'i).

\subsubsection{Etiqueta 1 --- \texttt{ingreso\_alto}}

\begin{cajainfo}{¿El alquiler gener\'o un ingreso alto?}
  \textbf{C\'alculo:} ingreso $\geq$ percentil 75 dentro de su categor\'ia.\\[3pt]
  \textbf{Valor:} detecta los alquileres m\'as rentables comparando
  dentro del mismo g\'enero, sin mezclar categor\'ias de precios distintos.\\[3pt]
  \textbf{Aplicaci\'on:} pricing din\'amico y campa\~nas por g\'enero.
\end{cajainfo}

\begin{lstlisting}[caption={C\'alculo de \texttt{ingreso\_alto}}]
umbral = df.groupby("categoria_nombre")["ingreso"].transform(
    lambda x: x.quantile(0.75)
)
df["ingreso_alto"] = (df["ingreso"] >= umbral).astype(int)
\end{lstlisting}

\subsubsection{Etiqueta 2 --- \texttt{pelicula\_popular}}

\begin{cajainfo}{¿La pel\'icula tiene alta demanda?}
  \textbf{C\'alculo:} \texttt{veces\_alquilada\_pelicula} $>$ mediana del cat\'alogo.\\[3pt]
  \textbf{Valor:} identifica los t\'itulos que m\'as tr\'afico generan
  y que conviene mantener siempre en stock.\\[3pt]
  \textbf{Aplicaci\'on:} gesti\'on de inventario y vitrina.
\end{cajainfo}

\begin{lstlisting}[caption={C\'alculo de \texttt{pelicula\_popular}}]
umbral = df["veces_alquilada_pelicula"].median()
df["pelicula_popular"] = (
    df["veces_alquilada_pelicula"] > umbral
).astype(int)
\end{lstlisting}

\subsubsection{Etiqueta 3 --- \texttt{genero\_alto\_valor}}

\begin{cajainfo}{¿El g\'enero es de alto valor econ\'omico?}
  \textbf{C\'alculo:} \texttt{precio\_promedio\_genero} $\geq$ percentil 75 entre g\'enero.\\[3pt]
  \textbf{Valor:} se\~nala los g\'eneros que histricamente generan m\'as
  ingresos y en los que vale la pena invertir.\\[3pt]
  \textbf{Aplicaci\'on:} estrategia de adquisici\'on de contenido y licencias.
\end{cajainfo}

\begin{lstlisting}[caption={C\'alculo de \texttt{genero\_alto\_valor}}]
umbral = df["precio_promedio_genero"].quantile(0.75)
df["genero_alto_valor"] = (
    df["precio_promedio_genero"] >= umbral
).astype(int)
\end{lstlisting}

\section{Conclusiones}
% =============================================================================

\begin{enumerate}[leftmargin=*, label=\textbf{\arabic*.}]

  \item La preparaci\'on y limpieza de datos es una etapa fundamental
        en cualquier proyecto de Machine Learning. Sin datos correctos
        y bien estructurados, los modelos no pueden aprender patrones
        confiables, independientemente del algoritmo que se utilice.

  \item Separar el proyecto en m\'odulos con responsabilidades claras
        (consultas, datos, etiquetas) facilita su mantenimiento y permite
        que distintos miembros del equipo trabajen en paralelo sin
        afectar el trabajo de los dem\'as.

  \item La definici\'on de etiquetas debe estar siempre guiada por el
        contexto del negocio. Una etiqueta que no responde a una pregunta
        \'util para la organizaci\'on no aporta valor, aunque el modelo
        la aprenda perfectamente.

  \item El uso de bases de datos NoSQL como MongoDB impone retos
        adicionales de calidad de datos, ya que los documentos pueden
        tener estructuras distintas. Es necesario anticipar y manejar
        estos casos antes de procesar la informaci\'on.

  \item La Inteligencia de Negocios y el Machine Learning son
        complementarios: BI identifica los patrones y tendencias del
        pasado, mientras que ML permite predecir comportamientos futuros
        y automatizar decisiones basadas en datos.

\end{enumerate}

% =============================================================================
\section{Recomendaciones}
% =============================================================================

\begin{enumerate}[leftmargin=*, label=\textbf{\arabic*.}]

  \item \textbf{Validar siempre la calidad de los datos antes de entrenar.}
        Antes de usar cualquier dataset para Machine Learning, es recomendable
        revisar valores nulos, duplicados y formatos inconsistentes. Un paso
        de exploraci\'on inicial (EDA) ahorra errores costosos m\'as adelante.

  \item \textbf{Documentar las decisiones de negocio que definen las etiquetas.}
        Los umbrales utilizados (percentil 75, mediana, etc.) deben quedar
        registrados y justificados, ya que son decisiones que pueden cambiar
        seg\'un las necesidades de la organizaci\'on.

  \item \textbf{Mantener los m\'odulos desacoplados.}
        Es buena pr\'actica que cada archivo tenga una sola responsabilidad
        y que los cambios en uno no obliguen a modificar los dem\'as.
        Esto aplica tanto en proyectos de BI como en desarrollo de software
        en general.

  \item \textbf{Versionar el dataset etiquetado.}
        Guardar una copia del CSV con la fecha de generaci\'on permite
        comparar resultados entre distintas versiones del modelo y reproducir
        experimentos anteriores si es necesario.

  \item \textbf{Considerar m\'etodos de validaci\'on cruzada al entrenar.}
        Cuando se pase a la etapa de entrenamiento, se recomienda usar
        t\'ecnicas como \textit{k-fold cross-validation} para obtener
        estimaciones m\'as robustas del rendimiento de los modelos.

\end{enumerate}
